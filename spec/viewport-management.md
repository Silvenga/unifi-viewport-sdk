# Viewport Management API

> Observed from network captures between a UNVR controller (Protect 7.1.83 / UniFi OS 5.1.19) and a UP Viewport
> (firmware `1.4.33`).

## Overview

An HTTP/TLS 1.3 server running on the device on port 8080 (appears to be hardcoded). The server appears to generate TLS
certificates on factory reset.

There might be a future bug in TLS certification generation in the future with current firmware the expiration date can
get really close to 2026 when the device is cold factory reset. It appears to sync to some time source after boot.

If reset by the console, it seems to generate certificates with an expiration far into the future (suggesting the
device's internal clock is used during certificate generation before the time sync).

The default TLS certification is `CN=UI RSA; O=UI, RSA 2048-bit, 25-year validity`. In my testing,
`ECDSA P-256; CN=localhost` seems perfectly fine.

## Controller-Facing Endpoints

These are the endpoints the Protect controller calls on the device via `POST https://{device}:8080/{path}` without
server certificate verification.

| Path            | Method | Purpose                              | Auth                            |
|-----------------|--------|--------------------------------------|---------------------------------|
| `/api/adopt`    | POST   | First-time adoption push             | `username` + `password` in body |
| `/api/readopt`  | POST   | Re-adoption (device already adopted) | `username` + `password` in body |
| `/api/info`     | POST   | Query device info (firmware, MAC)    | `username` + `password` in body |
| `/api/support`  | POST   | Download support file from device    | `username` + `password` in body |
| `/api/snapshot` | POST   | Request a snapshot from device       | `username` + `password` in body |
| `/api/locate`   | POST   | Locate (blink/beep) the device       | `username` + `password` in body |

### Authentication

The password can be changed by the controller after adoption (via `changeUserPassword` over UCP4)..

### Adoption Token

The token is a 32-character random alphanumeric string, generated by the controller when the user clicks "Adopt". The
token is sent to the device in the `/api/adopt` POST body, then the device presents it via the `x-token` header on the
WebSocket connection.

It appears that if the device doesn't respond to this adoption, the controller will keep trying (marking the device as
offline in the console).

Returning a 400 causes the console to raise an error with adoption.

### `POST /api/adopt`

The controller does not appear to parse the body - it only checks for a 200 status code. On an official ViewPort, a text
response is returned (`Successful`, need to confirm).

## Device-Side Endpoints (not called by the controller?)

### `POST /api/settings`

Observed in a network capture, sent by `curl/7.74.0` (not the Protect application). This only occurs when the firmware
is set to 1.0.0 (invalid firmware?).

The controller will keep on hitting this route.

### Request

```json
{
  "username": "ubnt",
  "password": "ubnt",
  "device": {
    "adb": true
  }
}
```

| Field        | Type   | Value    | Notes                                                                                                                                                    |
|--------------|--------|----------|----------------------------------------------------------------------------------------------------------------------------------------------------------|
| `username`   | string | `"ubnt"` | Management username                                                                                                                                      |
| `password`   | string | `"ubnt"` | Management password                                                                                                                                      |
| `device`     | object | `{...}`  | Device settings                                                                                                                                          |
| `device.adb` | bool   | `true`   | Feels obvious, but I haven't tested this. This might enable remote ADB. SSH is reported as disabled (I haven't tried TBH), remote ADB might be possible. |

### Response

```
HTTP/1.1 200 OK
Content-Type: text/plain
Date: Sun, 28 Jun 2026 00:48:46 GMT
Connection: keep-alive
Content-Length: 0
```

## TLS Notes

- Port 8080 uses TLS 1.3 (self-signed certs accepted).
